# TRUE 模拟实验改进报告

## 总述

当前实验已经具备一个可运行的比较框架，能够在统一任务域下比较 `TRUE`、`Blind` 和 `MOO` 三类协作机制，并且已经初步复现了以下核心现象：

1. `TRUE` 在总体效用和安全性上优于 `Blind`。
2. `TRUE` 在长期效用和新人融入上优于 `MOO`。
3. `MOO` 更容易出现分配集中和新人排斥。

但这套实验仍然属于“机制验证型原型”，离“严谨、全面、可用于稳定结论”的实验体系还有明显距离。下一阶段不建议盲目扩展到更多任务领域，也不建议引入表面复杂度更高但不可验证的伪真实工具。更合适的方向，是在当前桥梁设计抽象场景下，把“场景差异”系统地编码进概率模型、观测模型和效用模型中，使实验从“单点验证”升级为“参数化场景族下的机制比较”。

这意味着后续改进的重点不是换领域，而是定义多个有清晰统计含义的情境模式，例如：

- 安全优先型场景：效用与事故风险强烈非对称。
- 探索有利型场景：高方差实体在早期具有潜在价值。
- 信任误导型场景：高信任表征与真实质量在局部阶段出现背离。
- 审核延迟型场景：真实质量反馈晚于表面反馈到达。
- 分工互补型场景：局部最优个体组合不等于全局最优任务团队。

这样做的好处是，实验不再只是回答“某组参数下 TRUE 是否更好”，而是开始回答“在什么类型的环境中 TRUE 更强、为什么更强、边界在哪里”。

## 1. 改进目标

下一阶段实验建议围绕四个目标展开：

1. 从单一场景升级为参数化场景族。
2. 从启发式指标升级为可明确定义、可重复计算的指标体系。
3. 从单点结果升级为敏感性分析与稳健性分析。
4. 从“结论导向的调参”升级为“预先定义机制，再统一比较”。

在这一阶段，真实工具接入不是重点。重点是让当前模拟更像一类严格定义的生成模型。

## 2. 核心问题诊断

### 2.1 当前场景过于单一

现有实验本质上只定义了一个任务环境：

- 模块成功概率近似为 `capability × difficulty`
- 致命模块额外引入固定事故概率
- 效用函数固定为 `+100 / +60 / -200`
- 观测层固定为 `80% 机评 + 20% 人评`
- A8 操纵模式固定

这种设定可以跑通，但不能区分以下几类重要情况：

- 高风险行业与一般任务场景的差异
- 观测质量高与观测质量差的差异
- 早期探索收益高与探索收益低的差异
- 信任反馈真实与信任反馈被污染的差异

### 2.2 概率模型还不够丰富

当前成功模型相对简单，主要缺少：

- 模块之间的误差传播
- 队伍组合对成功率的影响
- 环境状态变化对成功率的影响
- “表面质量”与“真实质量”的双层生成机制

尤其是文档中最重要的 Goodhart 问题，目前只实现了部分现象，没有形成一个独立可控的“操纵型观测模型”。

### 2.3 指标与机制没有完全对齐

目前一些指标仍然偏启发式，例如：

- `trust_gini`
- `collapse_index`
- `A8 trust-quality correlation`

这些指标有分析价值，但还没有形成严谨、固定的实验定义。后续如果继续改模型，这些指标需要先被正式化，否则难以比较不同版本实验结果。

## 3. 总体改进原则

下一阶段建议坚持以下原则：

1. 不换任务域，先把任务域内的生成模型做全。
2. 不追求“更像现实”的表面复杂，而追求“更可解释、更可验证”的统计复杂。
3. 每增加一种场景差异，都必须明确它对应的是哪一类概率模型变化。
4. 每增加一个指标，都必须给出精确定义、计算方法和解释意义。

## 4. 场景族设计建议

建议把实验环境抽象成若干“场景模板”，每个模板只改一组生成机制，而不是同时乱改多个因素。下面是推荐的第一批场景族。

### 4.1 基准场景 Baseline

这是当前实验的标准化版本，用作所有后续比较的参考组。

特征：

- 成功率由 `capability × difficulty` 决定
- 固定事故惩罚
- 观测误差较低
- A8 有中等操纵能力

作用：

- 提供与当前结果连续可比的主实验基线

### 4.2 安全高惩罚场景 Safety-Critical

特征：

- 致命模块占比更高
- 致命错误惩罚从 `-200` 调高到 `-400` 或更高
- 低能力承担致命模块时事故概率更高
- 成功率对能力阈值更敏感

核心问题：

- 当安全约束极强时，TRUE 的优势是否进一步扩大
- MOO 是否更容易因为局部高信任而造成灾难性分配

### 4.3 探索收益场景 Exploration-Beneficial

特征：

- 某些新 Agent 的真实能力高，但先验完全未知
- 某些任务模块允许高方差实体在早期获得显著收益
- 探索成功后，系统整体效用提升很快

核心问题：

- `TRUE` 的 Thompson 采样是否能在这类环境中体现显著优势
- `MOO` 是否因为保守分配长期锁死在次优实体上

### 4.4 观测污染场景 Observation-Manipulated

特征：

- 真实质量 `q_true` 与表面质量 `q_surface` 分离
- 人类评审主要观察到 `q_surface`
- 自动评审主要观察到 `q_true`
- 部分时期 `q_surface` 与 `q_true` 正相关，部分时期负相关

核心问题：

- 是否能严格复现 Goodhart 型背离
- `TRUE` 是否因质量底线与约束过滤，在污染观测下保持更稳健

这类场景最适合替代当前对 A8 的简单手工偏置设定。

### 4.5 反馈延迟场景 Delayed-Feedback

特征：

- 某些模块在当轮只能看到表面评价
- 真实结果在若干轮后才暴露
- 系统需要在延迟反馈下更新信任

核心问题：

- TRUE 的遗忘因子与探索策略在延迟反馈环境中是否仍稳健
- MOO 是否因为过早优化错误信号而更加脆弱

### 4.6 局部冲突场景 Utility-Trust Misalignment

这是你提到的重点场景之一。

特征：

- 某些局部阶段中，高信任实体并不对应高即时效用
- 某些局部阶段中，高方差实体反而对应更高长期收益
- 也就是说，短期效用、当前信任均值和未来信息价值不完全一致

核心问题：

- TRUE 是否能在“短期看不划算、长期更优”的选择中保持优势
- MOO 是否因为过分依赖当前信任而丢失长期最优路径

这个场景不需要更换领域，只需要在成功率与效用函数中引入时间阶段性和局部反向关系即可。

## 5. 建议的概率模型升级

### 5.1 从单层成功模型升级为双层质量模型

当前模型：

- `P(success) = capability × difficulty`

建议升级为：

- 真实执行质量：`q_true`
- 表面可见质量：`q_surface`
- 最终观测信号：`q_obs`

例如：

- `q_true ~ Bernoulli(p_true)`
- `q_surface ~ Bernoulli(p_surface)`
- `q_obs = λ_auto * q_true + (1 - λ_auto) * q_surface`

这样可以自然表达：

- 格式很好但内容错误
- 表面评价高但真实质量差
- 自动评审发现问题但人类暂时被迷惑

### 5.2 从独立模块升级为条件模块模型

当前模块之间几乎独立，只受依赖约束影响。建议增加“误差传播”：

- 上游模块出错会降低下游模块成功率
- 但高能力下游执行者可以部分修复上游错误

例如：

- 若 `M1` 失败，则 `M6` 成功率额外乘以 `γ1 < 1`
- 若 `M5` 失败，则 `M4` 成功率下降

这会更符合工程任务结构，也更能体现“队伍组合质量”的重要性。

### 5.3 从个体成功率升级为团队成功率

当前分配主要是“每个模块选一个执行者”。下一阶段建议至少在模型上加入：

- 执行者能力
- 审核者能力
- 协调者能力

即一个模块不再只有主执行者，而可能由：

- 执行者
- 复核者
- 协调者

共同决定最终成功概率。

这会让 TRUE 的“多人多 Agent 协作”意义更完整，也能让 Blind 与 MOO 的差距更真实。

### 5.4 从静态环境升级为阶段环境

建议把 200 轮分成若干阶段：

- 早期：高不确定，探索价值高
- 中期：任务稳定，利用价值高
- 后期：需求变化，旧信任部分失效

这样可以自然测试：

- Thompson 采样是否在不同阶段有效
- 漂移检测是否必要
- MOO 是否更容易被旧信任锁定

## 6. 效用函数升级建议

当前效用函数简单明了，但可以再增强两点。

### 6.1 引入长期信息价值

当前效用只看当轮任务结果。建议增加一个分析层指标，而不是直接进入主优化：

- `information_gain`
- `future_value_proxy`

这样可以区分：

- 短期看亏，但帮助系统更快识别高能力新人的探索
- 短期看赚，但实际上加剧信任塌缩的保守分配

### 6.2 引入效用-安全权衡族

不要只用一个固定 `+100 / +60 / -200`。建议定义一组效用模板：

- 温和型：`+100 / +60 / -120`
- 标准型：`+100 / +60 / -200`
- 极端安全型：`+100 / +40 / -400`

这样就能看出 TRUE 的优势到底来自哪里：

- 是单纯安全导向的优势
- 还是在更一般效用结构下也成立

## 7. 观测与信任更新升级建议

### 7.1 将观测分为即时信号与延迟信号

建议把当前观测层拆成：

- 当轮信号：格式、速度、表面完整性
- 延迟信号：真实正确性、后验复核结果

这样能更真实地刻画“先被信任，后被揭穿”的路径。

### 7.2 引入评审者异质性

当前人类评审差异已经有一点点体现，但还不够。建议进一步引入：

- 只对某些模块评价更准
- 对 Agent 与人类存在系统偏见
- 在高压力场景中评价噪声上升

这能帮助验证 TRUE 是否在复杂观察环境下仍具稳健性。

### 7.3 正式引入漂移检测

建议按文档本意实现：

- 每若干轮比较当前信任分布与过去窗口的差异
- 如果差异大，则触发探索增强或信任重置

这不仅提升真实度，也有助于防止系统在需求变化后继续沿用过时信任。

## 8. 统计设计升级建议

### 8.1 主指标与辅指标分离

建议明确：

- 主指标：累积效用、致命错误、A9 首次被选延迟、选择塌缩指数
- 辅指标：平均质量、信任方差、A8 相关性等

主指标用于主结论，辅指标用于机制解释。

### 8.2 加入置信区间与稳健性检验

建议每个主指标都报告：

- 均值
- 标准差
- 95% 置信区间
- 效应量

同时增加：

- bootstrap 稳健性结果
- 非参数检验结果

### 8.3 进行参数敏感性分析

下一阶段最关键的一组实验，不是再加很多故事，而是系统扫参数。

建议先扫以下参数：

- 致命错误惩罚
- 任务难度分布
- 新 Agent 加入轮次
- A8 操纵强度
- 表面质量与真实质量相关性
- 观测层机评权重
- 遗忘因子 `ρ`

最终形成“参数热图”，而不是只给一组结果。

## 9. 实施优先级建议

### P0：必须先做

1. 写正式实验协议，固定指标定义。
2. 重构场景配置，把单场景改成参数化场景族。
3. 重构 A8 模型，引入 `q_true / q_surface / q_obs` 三层。
4. 增加参数敏感性分析框架。

### P1：第二阶段做

1. 加入阶段环境与反馈延迟。
2. 加入模块误差传播。
3. 重新定义 `trust_gini` 与 `collapse_index`。
4. 补充 bootstrap 与非参数检验。

### P2：进一步增强

1. 从单执行者模块升级为执行者-复核者-协调者联合模型。
2. 实现完整的漂移检测与自适应探索。
3. 把 TRUE 的分配器进一步向“候选组 + 约束过滤 + 求解”方向推进。

## 10. 建议的下一步交付物

如果按工程顺序推进，建议下一轮不是直接改全部代码，而是先完成三份文件：

1. `docs/protocol.md`
   - 固定假设、指标、显著性标准和实验流程

2. `docs/notes/scenario_family_design.md`
   - 定义 Baseline、Safety-Critical、Observation-Manipulated、Utility-Trust-Misalignment 等场景

3. `docs/notes/model_refactor_plan.md`
   - 说明如何把当前代码改造成“场景配置 + 统一仿真核心”

在此基础上，再开始改代码会更稳，也更不容易在实现中不断回头修定义。

## 结论

下一阶段最重要的改进方向，不是更换领域，也不是引入真实工具，而是把“场景差异”系统地映射到概率模型中。只有这样，实验才能从“在一个设定下 TRUE 看起来更好”，升级为“在一类可解释的环境条件下，TRUE 为什么更好、何时更好、优势边界在哪里”。

如果这一层做好了，后续不论是否接入真实工具，实验本身都会更有说服力。
